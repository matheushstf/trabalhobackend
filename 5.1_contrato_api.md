# 5.1. Contrato da API

## Objetivo

Este documento descreve o contrato da API do Sistema de Lanchonete, ou seja, o combinado entre Front-end e Back-end sobre rotas, formatos de request, formatos de response, autenticação, validações, regras de negócio e erros esperados.

Com este contrato, qualquer pessoa deve conseguir entender o que cada endpoint faz, saber quais dados enviar e receber, validar regras e testar a API usando Postman, Insomnia ou Swagger.

## Informações Gerais

| Item | Valor |
| --- | --- |
| Base URL local | `http://localhost:8080` |
| Prefixo principal | `/api` |
| Formato de dados | JSON |
| Autenticação | JWT Bearer Token |
| Header autenticado | `Authorization: Bearer <token>` |
| Swagger | `http://localhost:8080/swagger-ui.html` |
| OpenAPI JSON | `http://localhost:8080/v3/api-docs` |

## 5.2. Organização por Recurso

Os endpoints da API são organizados por recurso ou módulo funcional. A tabela abaixo apresenta uma visão consolidada para facilitar a navegação no contrato e a criação de coleções no Postman/Insomnia.

### `/auth` - Autenticação

| Método | Endpoint | Autenticação | Finalidade |
| --- | --- | --- | --- |
| `POST` | `/api/auth/register` | Pública | Cadastrar usuário cliente e retornar token JWT |
| `POST` | `/api/auth/login` | Pública | Autenticar usuário e retornar token JWT |

Observação: a API atual não possui endpoints de `refresh token` ou `logout`. Como o JWT é stateless, o Front-end encerra a sessão removendo o token armazenado localmente.

### `/usuarios` - Usuários e Perfil

| Método | Endpoint | Autenticação | Finalidade |
| --- | --- | --- | --- |
| `GET` | `/api/usuarios` | `ROLE_ADMIN` | Listar usuários cadastrados |
| `GET` | `/api/usuarios/{id}` | Token obrigatório | Consultar dados de um usuário |
| `PUT` | `/api/usuarios/{id}` | Token obrigatório | Atualizar nome do usuário |

Observação: o cadastro de usuário fica em `/api/auth/register`, pois cria o usuário e já retorna o token de autenticação.

### `/unidades` - Unidades e Cardápio por Unidade

| Método | Endpoint | Autenticação | Finalidade |
| --- | --- | --- | --- |
| `GET` | `/api/unidades` | Pública | Listar unidades ativas |
| `POST` | `/api/unidades` | Admin/Gerencial | Criar unidade |
| `GET` | `/api/unidades/{id}` | Pública | Consultar unidade por ID |
| `PUT` | `/api/unidades/{id}` | Admin/Gerencial | Atualizar unidade |
| `PATCH` | `/api/unidades/{id}/desativar` | Admin/Gerencial | Desativar unidade |
| `GET` | `/api/unidades/{id}/cardapio` | Pública | Listar produtos disponíveis no estoque da unidade |

### `/produtos` - Produtos/Cardápio

| Método | Endpoint | Autenticação | Finalidade |
| --- | --- | --- | --- |
| `GET` | `/api/produtos?page=1&limit=10` | Pública | Listar produtos cadastrados com paginação |
| `GET` | `/api/produtos/{id}` | Pública | Consultar produto por ID |
| `GET` | `/api/unidades/{id}/cardapio?page=1&limit=10` | Pública | Consultar produtos disponíveis por unidade |

Observação: a consulta geral fica em `/api/produtos`, mas a disponibilidade para venda deve ser consultada pelo cardápio da unidade, pois cada unidade possui estoque próprio.

### `/estoque` - Estoque por Unidade

| Operação | Como é atendida na API atual |
| --- | --- |
| Consulta de estoque disponível | `GET /api/unidades/{id}/cardapio`, retornando `quantidadeDisponivel` por produto |
| Saída de estoque | Automática ao criar pedido em `POST /api/pedidos` |
| Entrada/restauração de estoque | Automática ao cancelar pedido em `DELETE /api/pedidos/{id}`, quando aplicável |

Observação: no código atual não há controller público específico para `/api/estoque`. O estoque é controlado internamente pela regra de pedido e pelo cardápio por unidade.

### `/pedidos` - Pedidos

| Método | Endpoint | Autenticação | Finalidade |
| --- | --- | --- | --- |
| `POST` | `/api/pedidos` | Token obrigatório | Criar pedido, validar estoque, calcular total e baixar estoque |
| `GET` | `/api/pedidos?page=1&limit=10` | Admin/Atendente/Gerencial | Listar pedidos com filtros e paginação |
| `GET` | `/api/pedidos/{id}` | Token obrigatório | Consultar pedido por ID |
| `PATCH` | `/api/pedidos/{id}/status?status={status}` | Admin/Atendente/Cozinha/Gerencial | Atualizar status do pedido |
| `DELETE` | `/api/pedidos/{id}` | Admin/Gerencial | Cancelar pedido e restaurar estoque quando aplicável |

### `/pagamentos` - Pagamentos e Gateway Mock

| Método | Endpoint | Autenticação | Finalidade |
| --- | --- | --- | --- |
| `POST` | `/api/pagamentos/solicitar` | Token obrigatório | Solicitar pagamento do pedido via integração mock |
| `GET` | `/api/pagamentos/pedido/{pedidoId}` | Token obrigatório | Consultar pagamento associado a um pedido |

Regras principais: o pedido precisa estar `AGUARDANDO_PAGAMENTO`, só pode existir um pagamento por pedido e o retorno do gateway mock é salvo em `payloadRetorno`.

### `/fidelidade` - Programa de Pontos

| Método | Endpoint | Autenticação | Finalidade |
| --- | --- | --- | --- |
| `POST` | `/api/fidelidade/aderir` | Token obrigatório | Aderir ao programa de fidelidade |
| `GET` | `/api/fidelidade/meus-pontos` | Token obrigatório | Consultar saldo de pontos do usuário autenticado |
| `POST` | `/api/fidelidade/resgatar` | Token obrigatório | Resgatar pontos do usuário autenticado |

Observação: a API atual mantém saldo atual de pontos, mas não expõe endpoint de histórico de movimentações.

### `/promocoes` - Promoções

| Método | Endpoint | Autenticação | Finalidade |
| --- | --- | --- | --- |
| `POST` | `/api/promocoes` | Admin/Gerencial | Criar promoção |
| `GET` | `/api/promocoes/ativas?page=1&limit=10` | Pública | Listar promoções ativas com paginação |

## 5.4. Regras Mínimas do Contrato

Para evitar uma API inconsistente, o contrato segue os padrões abaixo.

### URLs no Plural

Os recursos principais usam nomes no plural:

| Recurso | Base path |
| --- | --- |
| Usuários | `/api/usuarios` |
| Unidades | `/api/unidades` |
| Produtos | `/api/produtos` |
| Pedidos | `/api/pedidos` |
| Pagamentos | `/api/pagamentos` |
| Fidelidade | `/api/fidelidade` |
| Promoções | `/api/promocoes` |

### IDs no Path

Consultas, atualizações e operações sobre um recurso específico usam o identificador no path:

```http
GET /api/produtos/{id}
GET /api/pedidos/{id}
PUT /api/usuarios/{id}
PUT /api/unidades/{id}
DELETE /api/pedidos/{id}
```

### Paginação em Listagens

Listagens aceitam os parâmetros `page` e `limit`.

Valores padrão:

| Parâmetro | Padrão | Regra |
| --- | --- | --- |
| `page` | `1` | mínimo `1` |
| `limit` | `10` | mínimo `1`, máximo `100` |

Exemplos:

```http
GET /api/produtos?page=1&limit=10
GET /api/pedidos?page=1&limit=10&canalPedido=APP&status=AGUARDANDO_PAGAMENTO
GET /api/unidades?page=1&limit=10
GET /api/unidades/1/cardapio?page=1&limit=10
GET /api/usuarios?page=1&limit=10
GET /api/promocoes/ativas?page=1&limit=10
```

### Status Codes Coerentes

| Status | Uso no contrato |
| --- | --- |
| `200 OK` | Consulta, listagem ou atualização concluída com sucesso |
| `201 Created` | Criação de recurso, como usuário, pedido, pagamento, unidade ou promoção |
| `204 No Content` | Operação sem corpo de resposta, como cancelamento de pedido |
| `400 Bad Request` | Erro de validação, JSON inválido, enum inválido ou parâmetro inválido |
| `401 Unauthorized` | Usuário não autenticado ou token inválido |
| `403 Forbidden` | Usuário autenticado sem permissão |
| `404 Not Found` | Recurso não encontrado |
| `409 Conflict` | Conflito ou regra de negócio, como e-mail duplicado, pagamento duplicado ou pedido em status inválido |
| `500 Internal Server Error` | Erro interno não tratado |

### Erro Padronizado

Todos os erros tratados retornam o mesmo formato JSON, variando apenas `status`, `error`, `message`, `path` e, quando houver validação de campo, `fieldErrors`.

```json
{
  "timestamp": "2026-07-18T10:30:00",
  "status": 400,
  "error": "Validation Error",
  "message": "Erro de validação nos campos",
  "path": "/api/produtos",
  "fieldErrors": [
    {
      "field": "page",
      "message": "deve ser maior ou igual a 1"
    }
  ]
}
```

## Autenticação e Autorização

A API usa autenticação via JWT. O token é obtido nos endpoints de cadastro ou login e deve ser enviado no header `Authorization` nas rotas protegidas.

Exemplo de header:

```http
Authorization: Bearer eyJhbGciOiJIUzI1NiJ9...
```

Rotas públicas:

| Método | Rota | Descrição |
| --- | --- | --- |
| `POST` | `/api/auth/register` | Cadastro de usuário |
| `POST` | `/api/auth/login` | Login |
| `GET` | `/api/unidades` | Listar unidades ativas |
| `GET` | `/api/unidades/{id}` | Buscar unidade |
| `GET` | `/api/unidades/{id}/cardapio` | Consultar cardápio da unidade |
| `GET` | `/api/promocoes/ativas` | Listar promoções ativas |
| `GET` | `/swagger-ui.html` | Interface Swagger |
| `GET` | `/v3/api-docs` | Especificação OpenAPI |

Rotas protegidas exigem token válido. Algumas rotas exigem perfis específicos, como `ROLE_ADMIN`, `ROLE_ATENDENTE`, `ROLE_COZINHA` ou perfil gerencial/administrativo.

## Padrão de Erro

Todos os erros tratados pela aplicação seguem o formato abaixo:

```json
{
  "timestamp": "2026-07-18T10:30:00",
  "status": 400,
  "error": "Validation Error",
  "message": "Erro de validação nos campos",
  "path": "/api/pedidos",
  "fieldErrors": [
    {
      "field": "unidadeId",
      "message": "ID da unidade é obrigatório"
    }
  ]
}
```

Principais status HTTP:

| Status | Quando ocorre |
| --- | --- |
| `200 OK` | Consulta, atualização ou operação concluída com sucesso |
| `201 Created` | Recurso criado com sucesso |
| `204 No Content` | Recurso cancelado/removido sem corpo de resposta |
| `400 Bad Request` | Erro de validação, JSON inválido, enum inválido ou parâmetro inválido |
| `401 Unauthorized` | Token ausente, inválido ou credenciais incorretas |
| `403 Forbidden` | Usuário autenticado sem permissão para a rota |
| `404 Not Found` | Recurso não encontrado |
| `409 Conflict` | Conflito ou regra de negócio |
| `500 Internal Server Error` | Erro interno não tratado |

## 5.5. Campos e Padrões Mínimos do Contrato

Nos contratos de pedido, o campo `canalPedido` é obrigatório e identifica por qual canal o pedido foi realizado.

Valores aceitos para `canalPedido`:

```text
APP, TOTEM, BALCAO, PICKUP, WEB
```

### Criação de Pedido

Endpoint:

```http
POST /api/pedidos
```

Request mínimo esperado pela API:

```json
{
  "unidadeId": 1,
  "canalPedido": "TOTEM",
  "itens": [
    {
      "produtoId": 10,
      "quantidade": 2
    }
  ]
}
```

Observação: nesta API, o pagamento é desacoplado do pedido. Por isso, a forma de pagamento não é enviada na criação do pedido; ela é enviada depois em `POST /api/pagamentos/solicitar` pelo campo `metodoPagamento`.

### Filtro de Listagem

Endpoint:

```http
GET /api/pedidos?canalPedido=APP&status=AGUARDANDO_PAGAMENTO
```

O filtro `canalPedido` é opcional na listagem, mas quando informado deve usar um dos valores aceitos pelo enum `CanalPedido`.

### Erro Quando `canalPedido` Não For Informado

Se `canalPedido` não for enviado na criação do pedido, a API retorna `400 Bad Request` com erro padronizado:

```json
{
  "timestamp": "2026-07-18T10:30:00",
  "status": 400,
  "error": "Validation Error",
  "message": "Erro de validação nos campos",
  "path": "/api/pedidos",
  "fieldErrors": [
    {
      "field": "canalPedido",
      "message": "Canal do pedido é obrigatório"
    }
  ]
}
```

### Erro Quando `canalPedido` For Inválido

Se `canalPedido` tiver valor inválido no corpo da requisição ou nos filtros da listagem, a API retorna `400 Bad Request` com erro padronizado:

```json
{
  "timestamp": "2026-07-18T10:30:00",
  "status": 400,
  "error": "Validation Error",
  "message": "Erro de validação nos campos",
  "path": "/api/pedidos",
  "fieldErrors": [
    {
      "field": "canalPedido",
      "message": "Valor inválido para o campo canalPedido. Valores aceitos: APP, TOTEM, BALCAO, PICKUP, WEB"
    }
  ]
}
```

Cadastra um novo usuário cliente e retorna um token JWT.

```json
{
  "nome": "Maria Cliente",
  "email": "maria@email.com",
  "senha": "123456",
```

Validações:

```json
{
```

| Status | Situação |
| --- | --- |
| `400` | E-mail já cadastrado |
| `400` | Consentimento LGPD não informado |
| `400` | Campos inválidos |

## Pedidos

### POST `/api/pedidos`

Cria um pedido para uma unidade, valida o estoque próprio da unidade, calcula o valor total, baixa o estoque e deixa o pedido com status `AGUARDANDO_PAGAMENTO`.

Autenticação: obrigatória.

Request:

```json
{
  "unidadeId": 1,
  "canalPedido": "APP",
  "itens": [
    {
      "produtoId": 1,
      "quantidade": 2
    },
    {
      "produtoId": 4,
      "quantidade": 1
    }
  ]
}
```

Validações:

| Campo | Regra |
| --- | --- |
| `unidadeId` | obrigatório; deve existir |
| `canalPedido` | obrigatório; deve ser um valor de `CanalPedido` |
| `itens` | obrigatório; deve conter ao menos um item |
| `itens[].produtoId` | obrigatório; produto deve existir |
| `itens[].quantidade` | obrigatória |

Regras de negócio:

| Regra | Descrição |
| --- | --- |
| Estoque por unidade | O produto precisa existir no estoque da unidade escolhida |
| Estoque suficiente | A quantidade solicitada não pode ser maior que a quantidade disponível |
| Baixa de estoque | Ao criar o pedido, a quantidade de cada item é subtraída do estoque da unidade |
| Status inicial | Todo pedido criado começa como `AGUARDANDO_PAGAMENTO` |
| Total | `valorTotal` é calculado pela soma dos subtotais dos itens |

Response `201 Created`:

```json
{
  "id": 10,
  "unidadeId": 1,
  "unidadeNome": "Unidade Centro",
  "usuarioId": 1,
  "usuarioNome": "Maria Cliente",
  "canalPedido": "APP",
  "status": "AGUARDANDO_PAGAMENTO",
  "valorTotal": 35.00,
  "itens": [
    {
      "id": 100,
      "produtoId": 1,
      "produtoNome": "X-Burguer",
      "quantidade": 2,
      "precoUnitario": 15.00,
      "subtotal": 30.00
    },
    {
      "id": 101,
      "produtoId": 4,
      "produtoNome": "Coca-Cola",
      "quantidade": 1,
      "precoUnitario": 5.00,
      "subtotal": 5.00
    }
  ],
  "dataHora": "2026-07-18T10:30:00"
}
```

Erros relevantes:

| Status | Situação |
| --- | --- |
| `400` | Pedido sem itens |
| `400` | Produto não disponível na unidade |
| `409` | Estoque insuficiente |
| `401` | Token ausente ou inválido |
| `404` | Unidade ou produto não encontrado |

### GET `/api/pedidos`

Lista pedidos com filtros opcionais.

Autenticação: exige perfil administrativo, atendente ou gerencial.

Query params opcionais:

| Parâmetro | Tipo | Exemplo |
| --- | --- | --- |
| `canalPedido` | `CanalPedido` | `APP` |
| `status` | `StatusPedido` | `PAGO` |
| `unidadeId` | `Long` | `1` |

Exemplo:


### GET `/api/pedidos/{id}`

Busca um pedido pelo ID.

Autenticação: obrigatória.

Response `200 OK`: objeto `PedidoResponseDTO`.

Erros relevantes: `401`, `404`.

### PATCH `/api/pedidos/{id}/status?status={status}`

Atualiza o status de um pedido.

Autenticação: exige perfil administrativo, atendente, cozinha ou gerencial.

Exemplo:

```http
PATCH /api/pedidos/10/status?status=EM_PREPARO
```

Regras de negócio:

| Regra | Descrição |
| --- | --- |
| Pedido cancelado | Não pode mudar status de pedido `CANCELADO` |
| Pedido entregue | Não pode mudar status de pedido `ENTREGUE` |
| Aguardando pagamento | Só pode ir para `PAGO` ou `CANCELADO` |

Response `200 OK`: objeto `PedidoResponseDTO` atualizado.

### DELETE `/api/pedidos/{id}`

Cancela um pedido.

Autenticação: exige perfil administrativo ou gerencial.

Regras de negócio:

| Regra | Descrição |
| --- | --- |
| Restauração de estoque | Se o pedido estiver `AGUARDANDO_PAGAMENTO` ou `PAGO`, o estoque dos itens é devolvido |
| Status final | Pedido passa para `CANCELADO` |

Response `204 No Content`.

## Pagamentos

### POST `/api/pagamentos/solicitar`

Solicita o pagamento de um pedido. O pagamento é desacoplado do pedido, possui registro próprio e armazena o payload de retorno da integração mock de pagamento.

Autenticação: obrigatória.

Request:

```json
{
  "pedidoId": 10,
  "metodoPagamento": "PIX"
}
```

Validações:

| Campo | Regra |
| --- | --- |
| `pedidoId` | obrigatório; pedido deve existir |
| `metodoPagamento` | obrigatório; deve ser um valor de `MetodoPagamento` |

Regras de negócio:

| Regra | Descrição |
| --- | --- |
| Status do pedido | Pedido deve estar `AGUARDANDO_PAGAMENTO` |
| Um pagamento por pedido | Se já existir pagamento para o pedido, a API retorna erro |
| Pagamento aprovado | Atualiza o pagamento para `APROVADO` e o pedido para `PAGO` |
| Pagamento recusado | Mantém o pedido aguardando pagamento e registra o retorno no pagamento |
| Payload de retorno | A resposta do gateway mock é armazenada em `payloadRetorno` |
| Idempotência | A restrição de um pagamento por pedido evita múltiplas cobranças para o mesmo pedido |

Response `201 Created`:

```json
{
  "id": 20,
  "pedidoId": 10,
  "valor": 35.00,
  "metodoPagamento": "PIX",
  "status": "APROVADO",
  "mensagem": "Pagamento aprovado",
  "payloadRetorno": "{\"status\":\"APROVADO\",\"mensagem\":\"Pagamento aprovado\"}"
}
```

Erros relevantes:

| Status | Situação |
| --- | --- |
| `400` | Pedido não está aguardando pagamento |
| `400` | Já existe pagamento para este pedido |
| `400` | Erro ao processar pagamento |
| `401` | Token ausente ou inválido |
| `404` | Pedido não encontrado |

### GET `/api/pagamentos/pedido/{pedidoId}`

Busca o pagamento associado a um pedido.

Autenticação: obrigatória.

Response `200 OK`:

```json
{
  "id": 20,
  "pedidoId": 10,
  "valor": 35.00,
  "metodoPagamento": "PIX",
  "status": "APROVADO",
  "mensagem": null,
  "payloadRetorno": "{\"status\":\"APROVADO\",\"mensagem\":\"Pagamento aprovado\"}"
}
```

Erros relevantes: `401`, `404`.

## Fidelidade

### POST `/api/fidelidade/aderir`

Adere o usuário autenticado ao programa de fidelidade.

Autenticação: obrigatória.

Regras de negócio:

| Regra | Descrição |
| --- | --- |
| LGPD | Usuário precisa ter consentimento LGPD |
| Único cadastro | Usuário não pode aderir duas vezes ao programa |
| Saldo inicial | Começa com `0` pontos |

Response `201 Created`:

```json
{
  "id": 1,
  "usuarioId": 1,
  "usuarioNome": "Maria Cliente",
  "pontosAcumulados": 0,
  "consentimentoPrograma": true
}
```

### GET `/api/fidelidade/meus-pontos`

Consulta os pontos do usuário autenticado.

Autenticação: obrigatória.

Response `200 OK`: objeto `FidelidadeResponseDTO`.

### POST `/api/fidelidade/resgatar`

Resgata pontos do usuário autenticado.

Autenticação: obrigatória.

Request:

```json
{
  "pontos": 10,
  "descricao": "Resgate de desconto"
}
```

Validações:

| Campo | Regra |
| --- | --- |
| `pontos` | obrigatório, mínimo `1` |
| `descricao` | opcional |

Regras de negócio:

| Regra | Descrição |
| --- | --- |
| Programa ativo | Usuário precisa estar cadastrado no programa |
| Consentimento | Programa precisa estar ativo para o usuário |
| Saldo suficiente | Usuário não pode resgatar mais pontos do que possui |

Response `200 OK`: objeto `FidelidadeResponseDTO` com saldo atualizado.

## Fluxo Crítico: Realizar Pedido + Solicitar Pagamento

Este é o fluxo principal esperado para o Front-end.

### 1. Login ou cadastro

Usar `POST /api/auth/login` ou `POST /api/auth/register` para obter o token JWT.

### 2. Consultar unidades

Usar `GET /api/unidades` para escolher a unidade.

### 3. Consultar cardápio da unidade

Usar `GET /api/unidades/{id}/cardapio` para exibir apenas produtos com estoque disponível naquela unidade.

### 4. Criar pedido

Usar `POST /api/pedidos` enviando unidade, canal e itens.

Resultado esperado: pedido criado com status `AGUARDANDO_PAGAMENTO`.

### 5. Solicitar pagamento

Usar `POST /api/pagamentos/solicitar` enviando `pedidoId` e `metodoPagamento`.

Resultado esperado:

| Resultado | Efeito |
| --- | --- |
| Pagamento aprovado | Pagamento fica `APROVADO` e pedido fica `PAGO` |
| Pagamento recusado | Pagamento fica `RECUSADO`; pedido não é liberado para preparo |
| Pagamento duplicado | API retorna erro porque já existe pagamento para o pedido |

## Checklist para Testes em Postman/Insomnia

1. Criar usuário em `POST /api/auth/register`.
2. Copiar o campo `token` da resposta.
3. Configurar `Authorization: Bearer <token>` nas rotas protegidas.
4. Listar unidades em `GET /api/unidades`.
5. Consultar cardápio em `GET /api/unidades/{id}/cardapio`.
6. Criar pedido em `POST /api/pedidos`.
7. Solicitar pagamento em `POST /api/pagamentos/solicitar`.
8. Consultar pagamento em `GET /api/pagamentos/pedido/{pedidoId}`.
9. Consultar pedido em `GET /api/pedidos/{id}`.
10. Testar erros esperados: estoque insuficiente, pagamento duplicado, token ausente e campos obrigatórios.

## Observações de Contrato

- Datas devem ser enviadas em formato ISO, por exemplo `2026-07-18T10:30:00`.
- Valores monetários usam número decimal, por exemplo `35.00`.
- O pagamento é desacoplado do pedido: consultar pedido e consultar pagamento são operações separadas.
- A API não deve permitir múltiplos pagamentos para o mesmo pedido.
- O cardápio deve ser consultado por unidade, pois cada unidade possui estoque próprio.
